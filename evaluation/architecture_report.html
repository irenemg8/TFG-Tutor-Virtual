#1 — Doble camino HTTP: orchestrator vs. ragMiddleware

CRÍTICO

Archivos: src/index.js:152-154 · src/interfaces/http/middleware/orchestratorMiddleware.js · src/rag/ragMiddleware.js

PROBLEMA

Dos implementaciones completas del mismo flujo de chat coexisten en producción. Un bug arreglado en el orchestrator NO se propaga al ragMiddleware (fallback). El ragMiddleware tiene ~800 líneas de lógica de negocio embebida que duplica (desincronizada) lo que hacen los agentes. Testear ambas rutas requiere doble esfuerzo.

SOLUCIÓN

Elevar USE_ORCHESTRATOR=1 a configuración permanente (eliminar la flag), añadir un test de humo E2E sobre el orchestrator y deprecar ragMiddleware.js en un sprint. El fallback real debe ser una respuesta 503 con mensaje claro, no otra implementación paralela.

graph LR subgraph ANTES["ANTES: dos caminos"] A1["POST /chat/stream"] --> B1{"USE_ORCHESTRATOR?"} B1 -->|sí| C1["orchestratorMiddleware\n+ 8 agentes"] B1 -->|no| D1["ragMiddleware\n~800 líneas"] end subgraph DESPUÉS["DESPUÉS: un camino"] A2["POST /chat/stream"] --> C2["orchestratorMiddleware\n+ 8 agentes"] C2 -->|error| E2["503 + log"] end

Beneficios

  • Elimina ~800 líneas de código muerto
  • Un único lugar para buscar bugs de chat
  • Suite de tests unificada

Riesgos / precauciones

  • Verificar 100% de tráfico en orchestrator antes de borrar
  • Añadir test E2E de flujo completo primero
  • Guardar ragMiddleware en rama archive si se necesita rollback

#2 — Degradación silenciosa del RAG por agotamiento de presupuesto

ALTO

Archivos: src/domain/agents/orchestrator.js:160-168 · src/infrastructure/search/hybridSearch.js · src/infrastructure/vectordb/embeddings.js

PROBLEMA

Si Ollama UPV tarda >8 s en generar el embedding (arranque frío habitual), retrievalAgent agota su presupuesto y el pipeline continúa sin augmentación RAG. El LLM responde sin contexto de dominio pero el usuario no lo sabe. No hay alerta, no hay indicador en el SSE, ni en los logs de negocio.

SOLUCIÓN

1. Añadir un hard-timeout explícito a la llamada de embedding (no heredado del presupuesto global).
2. Cuando se agota, emitir un evento SSE { rag_degraded: true } para que el frontend lo registre.
3. Subir la prioridad del warm-up de embeddings en el arranque de container.js (ya existe warm-up de Ollama pero no de embeddings).

#3 — Mapeo de ejercicio canónico duplicado en 3 sitios

ALTO

Archivos: src/infrastructure/llm/config.js · src/rag/ragMiddleware.js:52-73 · src/domain/agents/contextAgent.js

PROBLEMA

La lógica "ejercicio 2 usa el dataset del ejercicio 1" está en tres lugares distintos. Añadir ejercicio 8 requiere editar 3 archivos y acordarse de todos ellos. Ya ha causado bugs de inconsistencia entre rutas.

SOLUCIÓN

Crear CANONICAL_EXERCISE_MAP en src/infrastructure/llm/config.js y hacer que los tres sitios lo importen. Una sola fuente de verdad.

ANTES (3 copias)

// config.js
const map = { 2: 1, 6: 5 };

// ragMiddleware.js (línea 52)
if (exerciseId === 2) exerciseId = 1;

// contextAgent.js
const canonical = ex.id === 2 ? 1 : ex.id;

DESPUÉS (1 fuente)

// config.js
export const CANONICAL_EXERCISE_MAP = { 2: 1, 6: 5 };

// ragMiddleware.js, contextAgent.js, etc.
import { CANONICAL_EXERCISE_MAP } from '../config.js';
const canonical = CANONICAL_EXERCISE_MAP[id] ?? id;

#4 — HistorySummarizer opcional en lugar de obligatorio

ALTO

Archivos: src/domain/agents/contextAgent.js:31, 98-107 · src/container.js

PROBLEMA

Si el DI container no inyecta historySummarizer, el agente de contexto hace silenciosamente un windowing de los últimos N mensajes. Sesiones largas pierden contexto acumulado y el LLM genera respuestas inconsistentes. El bug no lanza error, simplemente el contexto se degrada.

SOLUCIÓN

1. Hacer el parámetro obligatorio: lanzar en constructor de contextAgent si historySummarizer es null.
2. Si se quiere mantener la funcionalidad de windowing como fallback, extraerla a un NullHistorySummarizer explícito que se pasa intencionalmente.
3. Añadir un test de integración que verifique inyección en container.

ANTES (fallo silencioso)

// contextAgent.js
constructor({ historySummarizer }) {
  this.summarizer = historySummarizer; // puede ser null
}
// ...
if (this.summarizer) {
  history = await this.summarizer.summarize(msgs);
} else {
  history = msgs.slice(-20); // fallback silencioso
}

DESPUÉS (explícito)

// contextAgent.js
constructor({ historySummarizer }) {
  if (!historySummarizer) throw new Error(
    'ContextAgent requires historySummarizer'
  );
  this.summarizer = historySummarizer;
}

// container.js — si quieres windowing explícito:
const summarizer = USE_SUMMARIZER
  ? new HistorySummarizer(llm)
  : new NullHistorySummarizer({ windowSize: 20 });

#5 — Dominio alcanza infraestructura: violación de la frontera hexagonal

MEDIO

Archivos: src/domain/agents/tutorAgent.js:29-30

PROBLEMA

Si debugLogger no se inyecta, tutorAgent hace un require('../../infrastructure/...') como fallback. Esto rompe la dirección de dependencia: domain → infrastructure. En tests, imposible mockear el logger.

SOLUCIÓN

Hacer el parámetro obligatorio (igual que #4) o usar un NullLogger inyectado intencionalmente. Nunca un require() condicional que atraviese la frontera.

ANTES

// tutorAgent.js
const logger = options.debugLogger
  ?? require('../../infrastructure/events/pipelineDebugLogger');

DESPUÉS

// tutorAgent.js
if (!options.debugLogger) throw new Error(
  'TutorAgent requires debugLogger'
);
const logger = options.debugLogger;

// container.js: siempre inyectar, incluso en tests
const logger = new NullLogger(); // implementa el puerto

#6 — Patrones de idioma hard-coded en cada guardrail

MEDIO

Archivos: src/infrastructure/guardrails/*.js (7 archivos) · src/domain/services/languageManager.js

PROBLEMA

Cada guardrail tiene sus propios patrones regex para es/val/en. Añadir un 4º idioma requiere editar 7 archivos. Los patrones están desincronizados entre guardrails (un fix en uno no se propaga a los demás).

SOLUCIÓN

Centralizar los patrones lingüísticos en languageManager.js (que ya existe) bajo una función getGuardrailPatterns(lang). Los guardrails reciben el objeto de patrones vía DI o lo piden al manager.

Impacto en el futuro: si se añade valenciano extendido o catalán, solo hay que tocar languageManager.js.

#7 — ChromaDB no-fatal en arranque: RAG puede iniciarse en modo degradado sin aviso

MEDIO

Archivos: src/container.js:118-144

PROBLEMA

Si las colecciones de ChromaDB están vacías, el container lanza un WARNING pero sigue inicializando. El servicio arranca aparentemente bien y el RAG semántico falla en silencio en cada request.

SOLUCIÓN

En producción: hacer el check FATAL (lanzar en container.initialize()).
En desarrollo: mantener warning + fallback BM25-only + log claro de qué colecciones faltan.
Añadir variable de entorno CHROMA_REQUIRED=false para entornos de desarrollo sin ChromaDB.

#8 — Modelos MongoDB todavía presentes en infrastructure/

BAJO

Archivos: src/infrastructure/persistence/mongodb/models/*.js (4 archivos)

PROBLEMA

La migración a PostgreSQL está completa pero los modelos Mongoose siguen en el repo. Confunden a lectores nuevos del código, y cualquier error de importación podría cargarlos accidentalmente.

SOLUCIÓN

Eliminar src/infrastructure/persistence/mongodb/ completo. Confirmar antes que no hay ningún require() activo que los use (grep lo confirma fácilmente).

⭐ Recomendación principal

#1 — Eliminar el doble camino HTTP es la acción de mayor retorno. Hoy cualquier bug en el flujo de chat puede estar solo en uno de los dos caminos y quedar invisible. El remedio es un test E2E del orchestrator completo + deprecar ragMiddleware.js.

Después, #2 — Degradación silenciosa del RAG es el problema con mayor impacto directo en la calidad pedagógica: el tutor puede responder sin contexto de dominio y nadie lo sabe. Un evento SSE rag_degraded + un hard-timeout en el embedding lo resuelve en pocas líneas.

  5. #2 — Degradación RAG visible (2-3 h, con test de embedding timeout)
