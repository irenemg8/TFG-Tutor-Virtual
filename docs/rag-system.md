# RAG System & Agent Orchestrator — Deep Dive

The pedagogical intelligence of the virtual tutor lives in the **domain** layer (`backend/src/domain/`). It is an **agentic, conditional RAG** system: every student message is classified, enriched with context retrieved from several sources, generated by an LLM, and validated by a chain of safety guardrails — all without ever revealing the solution.

This document covers the agent orchestrator and every domain service, how they connect, and the reasoning behind each design decision.

> The chat model is referred to as **qwen2.5**. The real-time pipeline monitor is out of scope.

## Overview

A standard RAG system retrieves documents and feeds them to an LLM. This system goes further on three axes:

- **Agentic** — the work is split into ten single-responsibility agents (`domain/agents/`) coordinated by an orchestrator over a shared context object.
- **Conditional** — a rule-based classifier sorts each message into one of nine types, and the pipeline routes each type to a different retrieval strategy and prompt.
- **Corrective & safe** — CRAG retries retrieval when results are poor, and a guardrail pipeline inspects every response, fixing or retrying anything pedagogically harmful before it reaches the student.

```
Student message
   │
   ▼
ContextAgent ─► (AcTrackerAgent ∥ InputGuardrailAgent) ─► ClassifierAgent ─► AcDetectorAgent
   │                                                                              │
   │  [exit: invalid exercise]        [exit: unsafe input]      [exit: greeting]  ▼
   │                                                            RetrievalAgent (RAG)
   │                                                                              │
   │                                            [exit: deterministic finish] ◄────┤
   ▼                                                                              ▼
PersistenceAgent ◄─ GuardrailAgent ◄─ PedagogicalReviewerAgent ◄─ TutorAgent (LLM call)
```

**Why non-streaming LLM calls?** The pipeline inspects the complete response before sending it, so guardrails can block or rewrite a turn that would leak the solution. A streamed token cannot be retracted. (The orchestrator can optionally stream tokens for latency and then send a correction frame if a guardrail rewrites the text.)

## The Orchestrator & the 10 Agents

**File:** `domain/agents/orchestrator.js`

`TutoringOrchestrator.process(request)` builds an `AgentContext`, splits a per-stage time budget, and runs the agents in order, honouring three early exits and always returning a populated context (with a localized fallback message on error).

### Agent contract

**Files:** `domain/agents/base/AgentInterface.js`, `agentRegistry.js`

Every agent extends `AgentInterface`:
- `name` — stable identifier.
- `async execute(context)` — reads inputs from the context, writes outputs back.
- `canSkip(context)` — gate to skip the agent (defaults to `false`).

`createAgentRegistry(deps)` instantiates all ten agents with their injected dependencies (repositories, LLM service, guardrail pipeline, classifier, RAG pipeline, security service, prompt builder, loggers) and returns them keyed by stage.

### The agents, in order

| # | Agent | File | Responsibility |
|---|---|---|---|
| 1 | **ContextAgent** | `contextAgent.js` | Loads the exercise, resolves/creates the interaction, windows the conversation history to `HISTORY_MAX_MESSAGES`, resolves the language, computes `cumulativeAnswer` and `loopState`, and summarizes older turns. Sets `fallthrough` if the exercise/tutor context is invalid. |
| 2 | **AcTrackerAgent** | `acTrackerAgent.js` | Aggregates the student's recurring alternative conceptions from past `resultados` and message evidence into `userACHistory` (top-N). Runs **in parallel** with the input guardrail. |
| 3 | **InputGuardrailAgent** | `inputGuardrailAgent.js` | Calls `securityService.analyzeInput()`; on an unsafe message sets `inputBlocked` and a localized redirect, ending the turn. |
| 4 | **ClassifierAgent** | `classifierAgent.js` | Runs the rule-based `classifyQuery()` over the message (and the last assistant text, for closed questions); writes `classification`. |
| 5 | **AcDetectorAgent** | `acDetectorAgent.js` | Deterministically matches AC patterns for the exercise, computes a per-element `turnVerdict`, and flags `stateMismatches` (student attributing the wrong state to an element). |
| 6 | **RetrievalAgent** | `retrievalAgent.js` | Runs the RAG pipeline (`runFullPipeline`): hybrid search + KG + CRAG, building the augmentation. `canSkip` on greeting/empty; honours the retrieval budget. |
| 7 | **TutorAgent** | `tutorAgent.js` | Builds the augmented system prompt (base prompt + pedagogical banners + RAG augmentation) and calls the LLM. Optional token streaming. |
| 8 | **PedagogicalReviewerAgent** | `pedagogicalReviewerAgent.js` | Deterministic, no-LLM repairs on the draft (strip premature confirmation, reframe definition requests, fix didactic explanations, enforce dataset style, fix code-switching). |
| 9 | **GuardrailAgent** | `guardrailAgent.js` | Delegates to `GuardrailPipeline.validate()`; writes `finalResponse`, the path taken, and maps violations onto metadata flags. Skipped on a deterministic finish. |
| 10 | **PersistenceAgent** | `persistenceAgent.js` | Saves the user and assistant messages with full metadata, writes the audit log, and emits an event. Always runs. |

### Early exits

1. **Invalid exercise** → `ContextAgent` sets `fallthrough`; the orchestrator returns and the request falls through to the legacy handler.
2. **Unsafe input** → `InputGuardrailAgent` sets `inputBlocked`; a localized redirect is persisted and returned.
3. **Deterministic finish** → after retrieval, `_shouldFinishDeterministically()` returns true when:
   - the exercise isn't already closed and there are no `stateMismatches`, **and**
   - the turn is `correct_good_reasoning` following a prior reasoned-correct turn, **or** the `cumulativeAnswer` is `closureReady` with no wrongly-named/excluded elements and the turn isn't a blocked type (greeting, off-topic, don't-know, or an explanation request).

   The orchestrator then builds a localized closure message ending in `<END_EXERCISE>` and skips the LLM, pedagogical reviewer and guardrails.

### Output cleanup

After the guardrails, the orchestrator strips any unauthorized `<END_EXERCISE>` token and normalizes whitespace (re-inserting spaces qwen2.5 sometimes drops after punctuation), then persists.

### Time budget

`ORCHESTRATOR_BUDGET_MS` (default 30 s) is divided per stage: **retrieval** 20 % (2–8 s), **guardrails** 10 % (1.5–5 s), **tutor** the remainder (≥ 8 s).

## The Shared Context (Blackboard)

**File:** `domain/agents/base/AgentContext.js`

A mutable object every agent reads from and writes to. Key fields:

- **Request inputs:** `userId`, `exerciseId`, `userMessage`, `interactionId`, `budgetMs`, `reqId`, `tokenStreamHandler`, `streamedText`.
- **Loaded context:** `exercise`, `exerciseNum`, `canonicalExerciseNum`, `correctAnswer[]`, `evaluableElements[]`, `history[]`, `lang`.
- **Loop state:** `loopState` (`prevCorrectTurns`, `consecutiveWrongTurns`, `totalAssistantTurns`, `tutorRepeating`, `studentFrustrated`, `lastAssistantQuestion`, `establishedFacts`, …).
- **Security & classification:** `inputSecurity`, `inputBlocked`, `classification`.
- **AC & answer state:** `detectedACs[]`, `turnVerdict`, `stateMismatches[]`, `cumulativeAnswer`, `userACHistory`, `exerciseAlreadyClosed`.
- **RAG & LLM:** `ragResult` (`augmentation`, `decision`, `sources`), `retrievalTimedOut`, `llmResponse`, `llmMessages`, `finalResponse`.
- **Guardrails:** `guardrailsTriggered` (per-guardrail flags), `guardrailPath`, `guardrailLlmRetries`, `guardrailSurgicalFixes[]`.
- **Timing & flags:** `timing` (`pipelineMs`, `ollamaMs`, `totalMs`, `firstTokenMs`), `deterministicFinish`, `fallthrough`, `fallbackUsed`.

**Thread safety:** Node is single-threaded and each request's pipeline is fully awaited, so a single context per request needs no locking.

| Parameter | Default | Meaning |
|---|---|---|

## Query Classification

**File:** `domain/services/rag/queryClassifier.js`

Every message is classified into one of nine types **before** any retrieval. The classifier is fully rule-based (regex + keyword matching over multi-language pattern dictionaries from `languageManager.js`) — no LLM.

### The 9 types

| Type | Example | Meaning |
|---|---|---|
| `greeting` | "Hola, ¿qué tal?" | Social greeting |
| `dont_know` | "No lo sé" | Student is stuck |
| `closedAnswer` | "Sí" / "No" | Yes/no reply to the tutor's closed question |
| `wrong_answer` | "R5" | Incorrect element selection |
| `correct_no_reasoning` | "R1, R2 y R4" | Right elements, no explanation |
| `correct_wrong_reasoning` | "R1, R2 y R4 porque forman un divisor de tensión" | Right elements, misconception as justification |
| `correct_good_reasoning` | "R1, R2 y R4 porque R3 está en abierto…" | Right elements, sound reasoning |
| `wrong_concept` | "R1 y R2 dado que forman un divisor de tensión" | Wrong elements driven by a misconception |
| `partial_correct` | "no pasa por R3" | Correct exclusions/partial proposals, incomplete |

### Decision order (first match wins)

1. Greeting opening + short message → `greeting`.
2. "Don't know" phrase → `dont_know`.
3. Question mark but no elements → `dont_know`.
4. Yes/no reply in a **closed-question** context (the tutor's last message was a closed question): diagnostic/meta question → `closedAnswer`; reasoning question → infer correct/wrong from the yes/no.
5. Extract elements, split into `proposed` vs `negated` (via the negation detector).
6. If `proposed` set equals the correct answer:
   - concepts + correct negations → `correct_good_reasoning`
   - reasoning with only state-description concepts (cortocircuito, abierto, …) → `correct_good_reasoning`
   - has concepts (potential misconception) → `correct_wrong_reasoning`
   - has reasoning → `correct_good_reasoning`
   - otherwise → `correct_no_reasoning`
7. Partially correct (some correct proposals/negations, set incomplete) → `partial_correct`.
8. Wrong elements + concept keywords → `wrong_concept`.
9. Otherwise → `wrong_answer`.

Output: `{ type, resistances[], proposed[], negated[], hasReasoning, concepts[] }`.

**Why rule-based?** Determinism (same input → same class), speed (< 1 ms vs seconds for an LLM), and no hallucinated categories. A correct answer with a suspicious concept is deliberately routed to `correct_wrong_reasoning` so it gets double-checked rather than blindly confirmed. State descriptions ("R3 en abierto") are distinguished from misconceptions ("divisor de tensión") so factual reasoning isn't penalized.

## Text Utilities

**Folder:** `domain/services/text/` — small, shared, single-source-of-truth helpers used by both the classifier (student input) and the guardrails (LLM output):

| Module | Purpose | Key exports |
|---|---|---|
| `accentNormalizer.js` | Accent-insensitive matching | `stripAccents`, `includesAsWord` |
| `elementExtractor.js` | Find element mentions | `extractResistances`, `extractMentionedElements` |
| `negationDetector.js` | Multi-language negation (pre-word, pre-phrase, post) | `detectNegationAround`, `isNegatedInContext`, pattern constants |
| `sentenceSplitter.js` | Sentence splitting (with/without trailing punctuation) | `splitSentences`, `splitSentencesKeepEnd` |
| `setComparison.js` | Element-set comparison | `sameSet`, `containsAll` |

Negation detection uses tight windows (≈15 chars before a single negation word, ≈30 for multi-word phrases, ≈25 after for post-negation, truncated at sentence boundaries) so a distant "no" doesn't wrongly flip an element ("R2 y R4. No pasa por R3" negates only R3).

## The RAG Pipeline (Routing & Augmentation)

**File:** `domain/services/rag/ragPipeline.js` (factory `createRagPipeline(deps)` → `{ runFullPipeline }`)

`runFullPipeline(userMessage, exerciseNum, correctAnswer, userId, evaluableElements, lang, options)` classifies, routes, retrieves and assembles the augmentation under a budget watchdog (`AbortController`), then appends the student's AC history.

### Routing

| Classification | Decision | Retrieval |
|---|---|---|
| `greeting` | `no_rag` | none |
| `dont_know` | `scaffold` | KG (basic concepts) |
| `closedAnswer` | `acknowledge_diagnostic` | none |
| `wrong_answer` | `rag_examples` | hybrid + CRAG if low score |
| `correct_no_reasoning` | `demand_reasoning` | hybrid |
| `correct_wrong_reasoning` | `correct_concept` | hybrid + KG |
| `correct_good_reasoning` | `rag_examples` | hybrid |
| `wrong_concept` | `concept_correction` | KG + hybrid |
| `partial_correct` | `rag_examples` | hybrid (+ KG) |

### Augmentation blocks

The augmentation appended to the system prompt is assembled from:

1. **`[RESPONSE MODE]`** — classification-specific instruction plus **intermediate feedback phrases**. For wrong/partial answers the LLM is told to open with a non-confirming phrase ("Hmm, no del todo…" / "Not quite…") so the response never starts by sounding correct. Phrases are language-aware. When the student answers a conceptual sub-question without naming elements, aggressive hints are softened so a conceptual reply isn't treated as wrong.
2. **`[PER-ELEMENT ANALYSIS]`** — per-element ground truth (correctly proposed / wrongly proposed / correctly excluded / wrongly excluded / missing), marked *internal, never reveal*. Includes the rule: if the student says an element "doesn't contribute" but it **is** in the answer, do not agree.
3. **`[DOMAIN KNOWLEDGE]`** — top knowledge-graph entries (concept, expert reasoning, AC, Socratic question), with a header reminding the LLM not to copy Socratic questions verbatim.
4. **Tone reference** — one retrieved student–tutor example (student side) for register.
5. **`[STUDENT HISTORY]`** — the student's top recurring AC errors across exercises.

A `[GUARDRAIL]` reminder is appended at the end of every augmentation.

## Knowledge Graph

**File:** `infrastructure/search/knowledgeGraph.js` (data in `data/knowledge-graph/…json`)

A structured store of circuit concepts. Each entry: `Node1`, `Relation`, `Node2`, `Expert reasoning`, `AC name`, `Description`, `Socratic Tutoring`. It encodes not just facts but common misconceptions and the Socratic question that addresses each.

- `searchKG(concepts)` — case-insensitive substring match over `Node1 + Relation + Node2` for the concept keywords the classifier found.
- `searchKGByAC(acId)` — look up entries by alternative-conception id.

**Why substring search, not vectors, at runtime?** The classifier emits exact concept terms ("divisor de tensión", "cortocircuito"); substring matching is precise for those, whereas semantic search could return related-but-wrong entries. (The KG is also ingested into ChromaDB for potential future use.)

## Hybrid Search

**Files:** `infrastructure/search/hybridSearch.js`, `bm25.js`, `infrastructure/vectordb/embeddings.js`, `chromaClient.js`

Two complementary retrievers fused with Reciprocal Rank Fusion.

### BM25 (keyword)
In-memory index per exercise, built from the student–tutor pairs at startup. Tokenization lowercases and splits on whitespace/punctuation, dropping tokens shorter than two characters. Scoring:

```
score(doc) = Σ IDF(t) × (tf × (k1+1)) / (tf + k1 × (1 − b + b × dl/avgDl))     k1=1.5  b=0.75
```

BM25 catches exact term matches (specific element names) but misses paraphrases.

### Semantic
The query is embedded to a 768d vector (`nomic-embed-text`) and matched in the per-exercise ChromaDB collection (`exercise_{n}`) by cosine similarity (`score = 1 − cosine_distance`, HNSW index). Semantic search catches conceptual similarity across different wording but can miss exact terms.

### RRF fusion

```
score(doc) = 1/(K + rank_bm25) + 1/(K + rank_semantic)     K = 60
```

Documents appearing in both lists rank higher. **Why RRF over a weighted sum?** BM25 scores (0–20+) and cosine similarity (0–1) live on incompatible scales; RRF uses only ranks, so no fragile normalization is needed. The top `TOP_K_FINAL = 2` survive.

Each stage emits start/end events on the `ragEventBus`, and ChromaDB queries accept an `AbortSignal` so retrieval respects the per-stage budget.

## CRAG — Corrective RAG

When `wrong_answer` retrieval returns a top score below `MED_THRESHOLD` (0.4), the pipeline reformulates and retries:

1. **Extract key entities** — element mentions (`R\d+`) plus matched concept keywords (any language), normalized to Spanish (the datasets are Spanish).
2. **Rebuild the query** — join the entities.
3. **Re-run hybrid search** with the focused query.

**Why entity extraction rather than an LLM rephrase?** Student circuit messages always revolve around specific components and concepts; extracting them focuses the search and discards filler, with no extra LLM latency.

## Alternative Conceptions (AC) Subsystem

ACs are deeply held student misconceptions ("current gets consumed", "voltage divider applies everywhere", "short circuit means no current"). Several services detect, track and address them.

| Module / Agent | Role |
|---|---|
| `domain/services/acRegistry.js` | `matchACs(patterns, proposed, negated, correct)` matches the turn against per-exercise AC patterns (from `data/contextos-ejercicios/tutorContext_por_ejercicio.json`), scored by match type. |
| `domain/services/kgRegistry.js` | `getStrategyForAC`, `getExpertReasoningForAC`, `getCatalogueEntry` — lazy lookup of strategy / reasoning / Socratic questions per AC id (from `alternative_conceptions.json` and the KG). |
| `AcDetectorAgent` | Runs `matchACs` for the current turn → `detectedACs`; also computes `turnVerdict` and `stateMismatches`. |
| `AcTrackerAgent` | Aggregates the student's recurring ACs across past results and messages → `userACHistory` (top-N), so the tutor can attend to recurring errors. |
| `domain/services/rag/cumulativeAnswer.js` | `computeCumulativeAnswer()` replays the classifier over the whole conversation to track answer state across turns: `namedCorrect`, `stillMissing`, `excluded`, `wronglyExcluded`, `wronglyNamed`, `reasoningConcepts`, `complete`, `closureReady`. This drives the deterministic finish. |
| `domain/services/rag/elementStates.js` | `deriveElementStates(netlist, expertReasoning)` infers each element's true state ("short"/"open"); `detectStateMismatch()` flags a turn that attributes the opposite state — which blocks a premature finish. |

The result-finalization endpoint (`POST /api/resultados/finalizar`) classifies the conversation's errors into a **closed list of AC ids** (LLM with JSON output, max 3), stored as `error_entries`.

## The Guardrail Pipeline

**File:** `domain/services/GuardrailPipeline.js`; adapters in `infrastructure/guardrails/`

The output safety system. `validate(response, ctx, opts)` runs guardrails, repairs deterministically where possible, and escalates to **at most one** consolidated LLM retry for surviving *critical* violations — all under a time budget. Worst case: 2 LLM calls per turn.

### Flow

1. **Phase A** — run every guardrail's `check()` in **parallel**. No violations → `primary_ok`.
2. **Phase B** — apply each violated guardrail's `surgicalFix()` (deterministic, no LLM); re-check. Clean → `surgical_ok`.
3. **Critical gate** — keep only violations whose id is in `CRITICAL_GUARDRAILS`. If none → `non_critical_only` (the response passes despite the residual). If budget too low → `budget_exhausted`. If no retry hints → `no_retry_hints`.
4. **Phase C** — append all critical retry hints to the system prompt and call the LLM **once**; re-check. Clean → `llm_retry_ok`.
5. **Phase D** — surgical fixes on the retry; final re-check → `llm_retry_plus_surgical` or `retry_failed_final_surgical`.

### Critical set

```
solution_leak · false_confirmation · premature_confirmation · state_reveal
complete_solution · repeated_question · adherence · settled_element_question
```

> **The "detected but not blocked" subtlety.** A guardrail can detect a violation and still let the response through if its id is **not** critical and no surgical fix applies (path `non_critical_only`). Always test behaviour through `GuardrailPipeline.validate()`, not a guardrail's `check()` alone.

### Guardrail catalogue

Each adapter implements `IGuardrail`: `id`, `severity`, `check()`, `surgicalFix()`, `buildRetryHint()`.

| `id` | Detects | Severity | Critical | Default profile |
|---|---|---|---|---|
| `language_drift` | Non-Latin script / wrong-language sentences | high | — | ✔ |
| `solution_leak` | Reveals the correct answer | med | ✔ | ✔ |
| `false_confirmation` | Confirms a **wrong** answer | high | ✔ | ✔ |
| `premature_confirmation` | Confirms a correct-but-unjustified answer | high | ✔ | legacy only |
| `complete_solution` | Confirms while the student wrongly named/excluded elements | high | ✔ | ✔ |
| `state_reveal` | Reveals an element's internal state / topology | med | ✔ | ✔ |
| `adherence` | Self-contradiction, multi-question, false premise/accusation | med | ✔ | ✔ |
| `repeated_question` | Repeats the previous Socratic question near-verbatim | med | ✔ | ✔ |
| `settled_element_question` | Re-asks about an already-settled element | med | ✔ | ✔ |
| `didactic_explanation` | Lectures instead of questioning | med | — | legacy only |
| `dataset_style` | Markdown / off-style formatting | low | — | legacy only |

**Profiles** (`GUARDRAIL_PROFILE`): **default** (8 hard-safety guardrails; pedagogical-style repairs are handled by the `PedagogicalReviewerAgent` instead) or **legacy** (all 11, for A/B comparison). Some guardrails are surgical-only (e.g. `dataset_style`); others are retry-only because no safe in-place rewrite exists (e.g. `repeated_question`, `settled_element_question` return `surgicalFix: null`).

**Why a pipeline instead of inline checks?** Parallel checks are fast; deterministic surgical fixes avoid an LLM round-trip; the critical gate prevents wasting the (single) retry on cosmetic issues; the budget guarantees the turn finishes in bounded time.

## Pedagogical Reviewer

**File:** `domain/agents/pedagogicalReviewerAgent.js`

Runs **before** the guardrails and makes deterministic, no-LLM repairs to the draft: strip a premature "¡Correcto!" on a partial answer, reframe a "define X" request into a Socratic question, remove textbook-style didactic explanations, enforce the dataset's plain style, and fix intra-sentence language mixing. Skipped on a deterministic finish or empty draft. Applied corrections are recorded in `pedagogicalCorrectionsApplied`.

This is where the style/pedagogy concerns that used to be guardrails now live, which is why the default guardrail profile omits `didactic_explanation` and `dataset_style`.

## LLM Integration & Prompt Building

### Adapters

**Files:** `infrastructure/llm/OllamaLlmAdapter.js`, `PoliGptLlmAdapter.js` (both implement `ILlmService`)

The provider is chosen by `LLM_PROVIDER`. Both expose `chatCompletion()` (non-streaming, used for guardrail inspection), streaming variants, a per-token callback variant, and `isHealthy()`. Generation options: temperature `0.4`, `num_ctx 8192`, `num_predict 220`, `keep_alive 60m`. The Ollama adapter posts to `/api/chat` and `/api/embed`; the PoliGPT adapter uses the OpenAI-compatible `/v1/chat/completions` and `/v1/embeddings`. `ILlmService` defines a `BudgetExhaustedError` so the orchestrator can fall back gracefully on timeout.

### Prompt building

**File:** `domain/services/promptBuilder.js` (`buildTutorSystemPrompt(ejercicio, lang)`)

The system prompt is assembled in blocks: Socratic rules (one question at a time; internal vocabulary that must never reach the student; element-naming constraints; the tutor's authority that the correct answer is ground truth; no false-premise questions), exercise info, objective, an **internal circuit-topology summary** parsed from the netlist (detecting shorts where `node1 === node2`), sanitized expert reasoning (sentences that would reveal the full answer are removed), relevant ACs, the normalized correct-answer set, and a version tag. Only populated blocks are emitted.

The `TutorAgent` then layers context-gated banners (loop-break, frustration, progress, recurring-AC, established facts) and the RAG augmentation on top, and splits stable vs. volatile content to help the model's KV cache.

## Loop Prevention

Implemented across `ContextAgent` (detection) and `TutorAgent` (banner injection):

- **Tutor repetition** — compares the last several tutor questions by word overlap; >50 % similarity flags repetition (catches A-B-A-B patterns a 2-message check misses) → `[ANTI-LOOP]` banner forcing a new, different question.
- **Student frustration** — multi-language patterns ("ya te lo he dicho", "I already told you") → `[STUDENT FRUSTRATED]` banner forcing empathy and progress.
- **Stuck** — when consecutive wrong turns exceed `MAX_WRONG_STREAK` (4) or total assistant turns exceed `MAX_TOTAL_TURNS` (16) → `[STUDENT IS STUCK]` banner forcing a strategy change with a concrete hint.
- **Loop override** — if the student has already answered correctly and the tutor is repeating, the turn is treated as a reasoned-correct close to end the exercise.

## History Summarization

**Files:** `domain/services/historySummarizer.js`, `NullHistorySummarizer.js`

To preserve context in long sessions without sending the whole transcript, turns that fall out of the recent `HISTORY_MAX_MESSAGES` window are folded into a rolling summary, cached per interaction and extended incrementally with a short LLM call. `NullHistorySummarizer` is a no-op used in dev/tests (recent window only).

## Observability

- **`ragEventBus.js`** — a singleton `EventEmitter`. Every significant step emits an envelope `{ requestId, timestamp, event, status, data }`. Used internally by the pipeline.
- **`jsonAuditLogger.js`** — when `AUDIT_LOG=1`, appends one JSON line per event to daily files under `logs/audit/` (buffered, flushed on a short interval and on shutdown). This is the data source for offline evaluation.
- **`pipelineDebugLogger.js`** — when `DEBUG_PIPELINE=1`, prints `[TRACE]` lines and mirrors them to the audit log, tracking per-request budget, LLM calls, guardrail checks, surgical fixes and stages.

Each assistant message is also persisted with rich `MessageMetadata` (classification, decision, guardrails, timing, detected ACs, guardrail path, surgical fixes), so analytics and exports work directly from PostgreSQL.

## The Legacy Linear Pipeline

`backend/src/rag/` (and `src/utils/promptBuilder.js`) hold the pre-refactor implementation: a single `ragMiddleware` that runs classification, security, retrieval, the LLM call and guardrails inline. It is **kept connected as a fallback** (it serves the chat request when `USE_ORCHESTRATOR` is off) and exists for A/B comparison against the orchestrator — not dead code. Both paths share the same domain services, so behaviour stays consistent; the orchestrator adds parallelism, explicit early exits, per-stage budgets and richer observability.

For multi-language details see [language-detection.md](language-detection.md); for diagrams see [architecture-diagrams.md](architecture-diagrams.md); for the data model and routes see [backend.md](backend.md).
