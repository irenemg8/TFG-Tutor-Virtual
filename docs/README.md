# Virtual Tutor — Project Documentation

An intelligent tutoring system for **electrical circuit analysis**. Students solve exercises by identifying which elements (resistances) contribute to a circuit property — for example, which resistances form a voltage divider. Instead of giving answers, the system uses **Socratic questioning**: it guides students to discover the solution themselves through targeted questions about circuit concepts.

The pedagogical core is an **agentic, conditional RAG (Retrieval-Augmented Generation) pipeline**. Every student message is classified, enriched with context retrieved from multiple sources (expert conversation examples, a domain knowledge graph, and the student's own error history), generated by an LLM, and then validated by a chain of **safety guardrails** that prevent the tutor from leaking the solution or confirming a wrong answer.

The codebase follows a **hexagonal architecture** (ports & adapters): the pedagogical logic lives in a framework-agnostic *domain* layer, while databases, the LLM, the vector store and HTTP are *infrastructure* adapters plugged in through *ports*. This makes the tutoring logic testable in isolation and lets implementations (PostgreSQL, the LLM provider, the guardrail set) be swapped without touching the core.

The three layers map directly to folders under `backend/src/`:

| Layer | Folder | Responsibility | Depends on |
|---|---|---|---|
| **Interfaces** | `interfaces/` | HTTP routes, chat middleware, SSE — translate the outside world into domain calls | Domain (via the container) |
| **Domain** | `domain/` | Entities, port interfaces, the agent orchestrator, the classifier, RAG and guardrail logic. **No framework imports.** | Nothing (pure logic) |
| **Infrastructure** | `infrastructure/` | Concrete adapters: PostgreSQL repositories, the LLM client, ChromaDB, BM25, guardrail implementations | Domain ports (implements them) |

Everything is wired together at startup by a single **dependency-injection container** (`backend/src/container.js`), which instantiates the adapters and injects them into the domain through the ports.

## Two Execution Paths

A chat request (`POST /api/ollama/chat/stream`) can be served by one of two pipelines, selected by the `USE_ORCHESTRATOR` feature flag. Both produce a streamed SSE response and persist the conversation; they differ in how the response is produced.

```
            POST /api/ollama/chat/stream
                        │
                        ▼
            ┌───────────────────────┐    USE_ORCHESTRATOR=1   ┌──────────────────────────────┐
            │ orchestratorMiddleware │──────── handles ───────►│  10-agent Orchestrator       │
            │  (gate)               │                         │  (domain/agents)             │
            └───────────┬───────────┘                         └──────────────────────────────┘
                        │ not enabled / not ready
                        ▼
            ┌───────────────────────┐    RAG_ENABLED ≠ false   ┌──────────────────────────────┐
            │ ragMiddleware (legacy) │──────── handles ────────►│  Linear RAG pipeline         │
            │  (gate)               │                         │  (kept for A/B comparison)   │
            └───────────┬───────────┘                         └──────────────────────────────┘
                        │ disabled / invalid input
                        ▼
            ┌───────────────────────┐
            │ ollamaChatRoutes      │  plain LLM fallback (no RAG, no guardrails)
            └───────────────────────┘
```

- **Orchestrator path (current, recommended):** ten single-responsibility agents run over a shared context object — load context → input security → classify → detect alternative conceptions → retrieve → generate → pedagogical repair → guardrails → persist. See [rag-system.md](rag-system.md).
- **Legacy linear path:** the original middleware that runs classification, retrieval, the LLM call and guardrails inline. It is **kept on purpose** (disconnected from the default flow) so the two designs can be compared on the same data — not dead code.

Both paths share the same domain services (classifier, RAG retrieval, guardrails, prompt builder, language manager), so behaviour stays consistent across them.

## Technology Stack

| Component | Technology | Purpose |
|---|---|---|
| Frontend | React 19, Vite 6, Tailwind 4, React Router 7 | Student-facing interface (chat, exercises, progress dashboard) |
| Backend | Node.js, Express 5 | API server, agent orchestrator, RAG pipeline, SSE streaming |
| Architecture | Hexagonal (ports & adapters) + DI container | Framework-agnostic pedagogical core |
| Database | PostgreSQL (`pg`), `connect-pg-simple` | Exercises, interactions, messages, results, users, sessions |
| Vector store | ChromaDB | Semantic search over student–tutor examples and the knowledge graph |
| LLM | Ollama **or** PoliGPT (selectable) | Tutor response generation — model: **qwen2.5** |
| Embeddings | nomic-embed-text (768 dimensions) | Text-to-vector conversion for semantic search |
| Keyword search | In-house BM25 (in-memory) | Exact-term retrieval, fused with semantic search via RRF |
| Auth | CAS OAuth2 (`simple-oauth2`) + dev bypass | University single sign-on; sessions stored in PostgreSQL |
| Language | `languageManager` (custom heuristics) | Multi-language support (Spanish, Valencian, English) |
| Evaluation | Python (custom metrics, PCA) | Retrieval and generation quality assessment, model comparison |

> **Note on the LLM provider.** The provider is chosen by the `LLM_PROVIDER` environment variable: `ollama` (local or a remote Ollama server) or `poligpt` (the UPV PoliGPT gateway). In both cases the documentation refers to the chat model as **qwen2.5**.

## Quick Start

For a complete, from-scratch setup see the [Deployment Guide](deployment-guide.md). In brief:

```bash
# 1. Install dependencies
cd backend  && npm install
cd ../frontend && npm install

# 2. Configure environment
#    Copy backend/.env.example → backend/.env and fill in:
#    - PG_CONNECTION_STRING (PostgreSQL)
#    - LLM_PROVIDER + model/endpoint settings
#    - SESSION_SECRET, CAS / DEV_BYPASS_AUTH

# 3. Start PostgreSQL (service) and ChromaDB
chroma run --host localhost --port 8000

# 4. Start the backend (runs DB migrations automatically on boot)
cd backend && npm start            # → http://localhost:3001

# 5. Ingest datasets + knowledge graph into ChromaDB (first time only)
node src/infrastructure/vectordb/ingest.js

# 6. Start the frontend
cd ../frontend && npm run dev      # → http://localhost:5173
```

On Windows, `deploy-hexagonal.ps1` automates steps 3–6 (PostgreSQL check, ChromaDB launch, `.env` patching, frontend build, nginx wiring). Use `verify.ps1` to confirm prerequisites before deploying.

## Key Design Decisions

### Why hexagonal architecture?

The pedagogical logic (how to classify a message, what to retrieve, which guardrails to apply) is the valuable, long-lived part of the system. Isolating it from databases, the LLM provider and HTTP means: the core can be unit-tested without a server or a database; PostgreSQL, the LLM, or the guardrail set can be swapped behind their ports; and the migration from the old MongoDB monolith was possible without rewriting the tutoring rules.

### Why a multi-agent orchestrator?

The original pipeline was a single long function. Splitting it into ten single-responsibility agents over a shared context (a *blackboard*) makes each step independently testable and observable, allows parallel stages (input security runs alongside error-history loading), and makes early exits explicit (greeting, blocked input, deterministic finish). The legacy linear pipeline is kept so the two can be compared empirically.

### Why rule-based classification?

An LLM classifier would be slower and non-deterministic — the same message could be classified differently on different calls, producing inconsistent tutoring. Rule-based classification (regex + keyword matching over multi-language pattern dictionaries) is instant, deterministic, transparent, and cannot hallucinate a category.

### Why hybrid search (BM25 + semantic) with RRF?

BM25 catches exact term matches (specific element names); semantic search catches conceptual similarity across different wording. Reciprocal Rank Fusion combines them using only ranks, so the two incompatible score scales never need normalizing.

### Why a guardrail pipeline instead of inline checks?

Each guardrail catches a different pedagogically harmful response (revealing the answer, confirming a wrong answer, prematurely confirming an unjustified answer, exposing internal element states, naming the element to inspect, repeating a settled question, drifting language, …). Running them as a pipeline allows them to be checked in parallel, fixed deterministically where possible (*surgical fixes*), and escalated to a single consolidated LLM retry only when a *critical* violation survives — all under a time budget.

### Why non-streaming guardrails?

The LLM is called in non-streaming mode so the full response can be inspected before it reaches the student. A streamed response would be sent token-by-token, making it impossible to retract a turn that leaks the solution. (The orchestrator can optionally stream tokens for latency, then send a correction frame if a guardrail rewrites the text.)

### Why PostgreSQL instead of MongoDB?

The data is inherently relational (users → interactions → messages → results → error entries) and benefits from foreign keys, transactional integrity and SQL analytics for the progress dashboard and exports. Sessions are stored in the same database, and conversation messages live in their own table rather than an embedded array, which makes per-message metadata and analytics straightforward.

## Glossary

| Term | Meaning |
|---|---|
| **AC (Alternative Conception)** | A common, deeply held student misconception about circuits (e.g. "current gets consumed"). The system detects, tracks and addresses ACs. |
| **Augmentation** | The extra context appended to the system prompt (classification hint, examples, knowledge, per-element analysis, student history, guardrail reminder). |
| **CRAG (Corrective RAG)** | A fallback that reformulates the query and retries retrieval when the first search returns poor results. |
| **Evaluable elements** | The set of circuit elements that count as a valid answer for an exercise (e.g. `["R1","R2","R3","R4","R5"]`). |
| **Guardrail** | A check that inspects the LLM response for a specific pedagogical hazard and either fixes it deterministically or triggers a retry. |
| **Knowledge Graph (KG)** | A structured store of circuit concepts, expert reasoning, alternative conceptions and Socratic questions. |
| **Orchestrator** | The agent that runs the ten pipeline agents over a shared context and returns the final response. |
| **Port / Adapter** | A *port* is a domain interface (e.g. `ILlmService`); an *adapter* is its infrastructure implementation (e.g. `OllamaLlmAdapter`). |
| **RRF (Reciprocal Rank Fusion)** | The method that merges BM25 and semantic rankings into one list. |
| **Surgical fix** | A deterministic, no-LLM correction a guardrail applies directly to the response. |
